## Codebase Patterns
- Pause/resume control paths should use a single global `AtomicBool` gate checked inside the capture loop, with API and CLI surfaces only toggling that shared state so pause semantics stay consistent across entry points.

- Event-driven capture triggers should bridge `NSWorkspace.didActivateApplicationNotification` through a C callback into a bounded Rust `mpsc` channel managed by an RAII listener guard, so app-switch captures fire immediately without leaking observers.

- E2E daemon integration tests should manage spawned `Child` processes with a Drop guard that kills and waits, so failures and panics cannot leak background daemons between tests.
- Menu bar API integrations should centralize `URLSession` polling in a SwiftUI `ObservableObject` refreshed by a `Timer`, so status/icon/menu state updates stay consistent across asynchronous health checks.
- Menu bar scaffolding should run as an AppKit accessory app with an `NSStatusItem`, poll `screencap status` on a short timer, and derive icon tint/menu enablement from daemon liveness (`running` green, `stopped` red, `unavailable` gray).
- Prune flows should resolve stored screenshot paths against the configured storage root, remove on-disk JPEGs before deleting database rows, and tolerate already-missing files so retention jobs stay idempotent while still reporting reclaimed bytes.
- MCP stdio servers should keep stdout strictly JSON-RPC protocol output and route diagnostics/logging to stderr so clients can parse responses deterministically.
- MCP transport tests should spawn the real CLI child process and exercise initialize/tools/list/tool-call flows over stdio, because line framing behavior differs across clients.
- Config loaders should separate where config lives from what config resolves to: accept explicit root/home inputs for tests, then create runtime directories from resolved storage and export paths after parsing.
- When FTS search text is assembled from multiple tables, use a standalone FTS index keyed by the primary row id and rebuild it from storage writes instead of relying on an external-content FTS table tied to a single base table.
- For Swift capture bridges, keep enumeration and capture on the same native backend, expose C-callable functions from `swift/Sources/`, and keep `mock-capture` JPEG generation in Rust so permissionless tests exercise real files.
- When the Swift bridge spans multiple source files, build the archive directly with `swiftc -emit-library -static` over all `swift/Sources/*.swift` inputs; `-emit-object -o` only works for a single Swift source.
- When resolving frontmost window titles from `CGWindowListCopyWindowInfo`, stop at the first layer-0 window for the active app PID; if it is untitled, propagate an empty string instead of falling through to another window from the same app.
- When one capture cycle spans multiple displays, stage every JPEG for that timestamp and persist the matching `captures` rows in one SQLite transaction; if any write fails, remove the new files so the screenshot tree and database cannot diverge.
- Daemon lifecycle should use a PID file under `~/.screencap/` that stores both `pid` and `started_at`; `start`, `stop`, and `status` should verify process liveness so stale PID files self-heal instead of blocking future runs.
- REST API read endpoints should open SQLite read-only and return empty/404 responses without creating `screencap.db`; read traffic must not mutate runtime state.
- CLI read commands should open SQLite read-only when possible, return helpful empty-state output without creating `screencap.db`, and label capture-count-based project breakdowns explicitly so terminal summaries do not overclaim time accuracy.
- Search and insight read surfaces should return typed records that already include their joined capture context, so API/UI callers can filter by app/timestamp and render screenshot metadata without issuing per-hit follow-up queries.
- Screenshot-serving endpoints should sanitize relative paths and walk them with `openat(..., O_NOFOLLOW)` so symlinks and traversal attempts cannot escape the screenshots root.
- OpenAI-compatible provider code should resolve default base URLs from the provider enum in one place and return structured response text plus optional token usage, so local backends that omit usage do not force fake cost data.
- When OpenAI-compatible providers expose usage cost metadata (such as OpenRouter’s `usage.cost`), pass it through to persisted `cost_cents`; providers that do not report cost should stay `None` rather than fabricating values.
- LLM JSON parsers should tolerate optional markdown code fences from model output, but they should still validate invariants like unique capture IDs before persisting so downstream jobs can trust the parsed batch.
- Extraction batches should persist atomically: write the `extraction_batches` summary/debug row, all per-frame `extractions`, the matching `captures` status/link updates, and the shared FTS refresh in one SQLite transaction; if parsing fails after an LLM call, still store the raw response before marking those captures failed.
- When synthesis needs both extraction batch summaries and frame-level evidence, add a typed storage helper that groups `extraction_batches` with joined `captures`/`extractions`; keep multi-table SQL in `StorageDb` so prompt builders stay focused on prompt shape.
- Daily synthesis should read typed hourly `Insight` rows from `StorageDb`, and on-demand entry points should check for an existing daily summary before opening an LLM provider so already-generated summaries stay readable without API credentials.
- For web shell scaffolding, use SvelteKit route files (`+layout.svelte`, `+page.svelte`) with shared navigation metadata in `src/lib/utils/nav.ts` and `adapter-static` fallback `index.html` so embedded static hosting supports deep links.
- For embedded web delivery, compile SvelteKit with `adapter-static` to `web/dist`, embed that directory via `rust-embed`, and have the Rust server fall back to `index.html` for non-API routes so deep links resolve without hash routing.
- Keep embedded web delivery on a dedicated npm build script (`build:embed`) called from `build.rs`, and support `SCREENCAP_WEB_DEV=1` to skip npm during Rust builds while still ensuring `web/dist/index.html` exists for SPA fallback serving.
- Timeline route pagination should read lightweight batches from `/api/captures`, hydrate only processed cards through `/api/captures/:id`, and trigger incremental loads with an intersection sentinel so infinite scroll works whether the page scrolls at window level or inside a container.
- Insights route components should normalize polymorphic synthesis payloads (`data` shape drift, missing fields, alternate key casing) into strict UI view-models before rendering, so empty/error states stay deliberate and the UI remains resilient to API evolution.
- Search route UIs should debounce query typing, gate stale async responses with request IDs, and keep app/project/time chips wired to server-side `/api/search` filters so relevance ranking remains authoritative.
- Settings route UIs should derive daemon liveness from `/api/health`, pair it with `/api/stats` telemetry for storage/capture panels, and always provide a first-class disconnected state with explicit CLI recovery steps.
- Markdown export commands should treat `--output` as a file for single-day exports but require a directory for multi-day `--last` ranges; generate `{date}.md` files per day and keep stdout export as the default only when no custom daily export path is configured.
- Prompt templates should load from `~/.screencap/prompts/*.txt` with default templates seeded once and existing custom files preserved, so prompt behavior is operator-tunable without recompiling.
- Capture persistence should treat each multi-display timestamp as one atomic unit: track JPEGs created in that cycle and remove them if any transactional DB write fails, so retries cannot inherit orphaned files.
- Timeline pagination fetch loops should maintain explicit `nextOffset` and `hasMore` state derived from each `/api/captures` page size, so infinite loading remains deterministic without relying on separate total-count queries.
- Mock LLM providers used by daemon E2E tests should synthesize schema-valid default JSON (extraction + rolling/hourly/daily summaries) when response queues are empty, so full scheduler startup can run without external API keys or brittle per-call fixture setup.
- Daemon runtime should spawn pipeline schedulers as independent shutdown-aware tasks, but treat missing/unsupported AI provider setup as a logged no-op so capture/API startup still succeeds while operator credentials are being configured.
- Writable SQLite connections should enable a busy timeout plus WAL mode, because the daemon runs capture, API, and synthesis/extraction schedulers concurrently and should absorb brief lock contention instead of failing writes.
- Rolling synthesis windows should be truncated to whole seconds before prompt construction and response validation, because the prompt format serializes RFC3339 timestamps without subsecond precision.
